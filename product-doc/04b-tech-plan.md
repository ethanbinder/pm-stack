# Tech Plan: Append PM Stack attribution footer to Atlassian content created by skills

**Author**: Ethan Binder

**Objective**: Wire the existing `Built with [Ethan's PM Stack](https://github.com/ethanbinder/pm-stack)` footer into Jira tickets that `/release` creates, and document the same convention for any future Confluence-creating flow. Credit travels with the artifact, not just the GitHub PR.

**PRD & Design Link**:

---

## Problem Statement
PM Stack PR bodies already carry the attribution footer (recipe at `skills/release/SKILL.md:148`). Jira tickets created by `/release` did not. Confluence pages don't get created by any PM Stack skill today, but if one ever does, it should follow the same convention. Without this, the attribution lives only on the GitHub PR — readers landing on the Jira ticket (or a future Confluence page) have no signal that the artifact was generated by a PM Stack skill.

## Changes Made
- `skills/release/SKILL.md` — Jira-mode step 2 (Ensure a ticket exists): added a sub-bullet that instructs the skill to append a two-line footer (`---` rule + `Built with [Ethan's PM Stack](...)`) to the `--description` body of every `jira create` call. Updated the surrounding bullet to reference `--description "<body>"` explicitly. Footer applies to ticket descriptions only, not to subsequent `jira issue comment add` calls (comments stay clean).
- `skills/release/SKILL.md` — Confluence-mode section: appended a forward-looking-convention bullet stating that any future flow (in `/release` or any PM Stack skill) that *creates* a Confluence page must append the same footer to the page body. Edits to existing pages are excluded so the footer doesn't stack.

The PR-body footer at `skills/release/SKILL.md:148` is unchanged. No other skills are modified — `/release` is the only skill that creates Atlassian content today.

## Risks
- **Atlassian markup variance.** The footer uses GitHub-flavored markdown link syntax (`[text](url)`). The modern Atlassian Cloud editor renders this correctly, but legacy on-prem Server installations may require wiki markup (`[text|url]`). For users on legacy Jira Server, the footer would render as literal markdown rather than a link. Acceptable trade-off: PM Stack's user base is overwhelmingly Cloud, and the footer text remains human-readable even if not hyperlinked.
- **No retroactive update.** Existing Jira tickets created by prior `/release` runs will not be back-filled with the footer. Only net-new tickets get the footer.
- No security implications. No data-handling changes. No breakage to the tech-plan URL recipe (PR #41) or the PR-body footer.
